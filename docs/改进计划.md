## 三、详细改进清单

### 3.1 性能优化

#### 改进项 #5: 向量化聚合计算

**当前问题**:
```rust
// crates/be-execution/src/exec_node.rs:258-263
// 逐行提取 scalar，性能极差
let values: Vec<ScalarValue> = (0..block.num_rows())
    .map(|i| col.scalar_at(i))
    .collect();
let agg_value = Self::compute_aggregate(&values, func);
```

**建议方案**:
- 使用批量操作直接处理 Vector 数据
- 避免中间 ScalarValue 转换
- 参考 Arrow 的 aggregate 实现

**实施步骤**:
1. 为每种 Vector 类型实现 `sum_batch()`, `count_batch()`, `avg_batch()` 等方法
2. 修改 AggregateExecNode 使用批量 API
3. 添加基准测试验证性能提升

**预期收益**: 聚合查询性能提升 **5-10x**

---

#### 改进项 #6: 向量化排序

**当前问题**:
```rust
// crates/be-execution/src/exec_node.rs:449-480
// 逐行比较，闭包捕获 block，无法 SIMD
indices.sort_by(|&a, &b| {
    for &(col_idx, ascending) in &order_by {
        let scalar_a = col.scalar_at(a);
        let scalar_b = col.scalar_at(b);
        // 逐行比较...
    }
});
```

**建议方案**:
- 使用列式比较器，一次比较整列
- 使用 `sort_unstable_by` + 向量化比较
- 考虑使用 `radix_sort` 对数值类型

**实施步骤**:
1. 为 Vector 实现 `compare_indices` 批量比较方法
2. 修改 SortExecNode 使用批量比较
3. 对数值类型使用基数排序

**预期收益**: 排序性能提升 **3-5x**

---

#### 改进项 #7: SIMD Bitmap 操作

**当前问题**:
```rust
// crates/types/src/bitmap.rs:214-227
// 手动实现位运算，未使用 SIMD
impl BitAnd for &Bitmap {
    fn bitand(self, rhs: &Bitmap) -> Bitmap {
        for i in 0..words {
            data.push(left & right); // 逐 word 操作
        }
    }
}
```

**建议方案**:
- 使用 `std::simd` (nightly) 或 `portable-simd`
- 一次处理 256/512 bits
- 使用 `count_ones` SIMD 指令

**实施步骤**:
1. 添加 `packed_simd` 或 `std::simd` 依赖
2. 实现 SIMD AND/OR/NOT 操作
3. 实现 SIMD `set_count()` 和 `iter_set_bits()`
4. 添加基准测试对比性能

**预期收益**: Bitmap 操作性能提升 **4-8x**

---

#### 改进项 #8: StringVector 零拷贝优化

**当前问题**:
```rust
// crates/types/src/vector.rs:228-246
// 每次 filter/slice 都重新分配和拷贝
pub fn filter(&self, selection: &Bitmap) -> Self {
    for idx in selection.iter_set_bits() {
        if let Some(s) = self.get(idx) {
            data.extend_from_slice(s.as_bytes()); // 每次都拷贝
        }
    }
}
```

**建议方案**:
- 使用偏移量引用而不是拷贝（类似 Arrow StringArray）
- 添加 `StringView` 类型，持有对原始数据的引用
- 延迟拷贝到必要时

**实施步骤**:
1. 设计 `StringView` 结构（offsets + data 引用）
2. 实现零拷贝 filter/slice
3. 在必要时触发拷贝（如数据生命周期结束时）

**预期收益**: 字符串过滤性能提升 **2-4x**，内存使用减少 **50-70%**

---

#### 改进项 #9: MemTable 结构优化

**当前问题**:
```rust
// crates/be-storage/src/tablet.rs:59-67
// 每行存储为一个 Block，内存开销大
pub struct MemTable {
    rows: BTreeMap<MemTableKey, Block>, // 每行一个 Block
    memory_size: u64,
    capacity: u64,
}
```

**建议方案**:
- 改为列式存储：`BTreeMap<MemTableKey, RowData>` 或直接列式
- 批量写入时合并数据
- 考虑使用 ART (Adaptive Radix Tree) 替代 BTreeMap

**实施步骤**:
1. 重新设计 MemTable 为列式结构
2. 实现批量插入优化
3. 添加内存精确统计
4. 考虑集成 ART 库（如 `art` crate）

**预期收益**: 内存使用减少 **30-50%**，写入性能提升 **2-3x**

### 3.2 类型系统重构

#### 改进项 #13: Vector 类型系统重构

**当前问题**:
```rust
// crates/types/src/vector.rs:5-20
// 13 个变体的 enum，每次 match 都要处理所有情况
pub enum Vector {
    Boolean(BooleanVector),
    Int8, Int16, Int32, Int64, Int128,
```

**建议方案**:

- 使用 trait `TypedVector` 定义通用接口
- 减少 match 分支数量

**实施步骤**:
1. 定义 `TypedVector` trait
2. 将 Vector 重构为分组 enum
3. 使用宏生成 trait 实现
4. 更新所有使用 Vector 的代码
5. 添加测试确保功能一致

**预期收益**: 代码减少 **30-40%**，维护性提升

### 3.3 并发和异步改进

#### 改进项 #11: 并发结构优化

**当前问题**:
```rust
// 多处使用 RwLock<HashMap>，高并发时性能瓶颈
tablets: RwLock<HashMap<u64, Arc<Tablet>>>,  // engine.rs:17
nodes: RwLock<HashMap<NodeId, BeNode>>,      // cluster.rs:142
```

**建议方案**:
- 使用 `DashMap`（并发 HashMap）替代
- 或使用 `left_right` crate 实现无锁读取
- 减少锁粒度

**实施步骤**:
1. 添加 `dashmap = "6"` 依赖
2. 替换 `RwLock<HashMap>` 为 `DashMap`
3. 更新所有访问代码
4. 添加并发基准测试

**预期收益**: 并发吞吐提升 **2-3x**

## 五、风险评估与缓解策略

### 高风险项

| 改进项 | 风险 | 缓解策略 |
|--------|------|---------|
| Vector 重构 | 影响范围大，可能引入 bug | 充分测试，逐步迁移 |
| Pipeline 异步化 | API 变更大，需要大量修改 | 分阶段实施，保持向后兼容 |
| Arrow 集成 | 依赖外部库，可能不兼容 | 先评估兼容性，逐步替换 |

### 中风险项

| 改进项 | 风险 | 缓解策略 |
|--------|------|---------|
| SIMD 优化 | 需要 nightly 或额外依赖 | 使用 portable-simd，保持兼容性 |
| 并发结构优化 | DashMap 可能有不同语义 | 充分测试并发场景 |
| 错误处理精细化 | 需要修改所有错误处理代码 | 使用工具辅助重构 |

### 低风险项

| 改进项 | 风险 | 缓解策略 |
|--------|------|---------|
| Clippy 修复 | 几乎无风险 | 自动修复 + 手动检查 |
| Default 实现 | 无风险 | 直接添加 |
| 清理未使用代码 | 无风险 | 直接删除 |
